01

What is a Dev Container?

A Dev Container is a Docker container configured specifically for development. Instead of installing runtimes, CLI tools, and extensions locally on each machine, your editor connects inside the container where everything is already set up.

The entire environment — language runtime, tools, extensions, settings — is defined as code and lives inside your project repository.

How it works

Your Machine
→
VS Code
→ connects to →
Docker Container
→
Your Project Files

Dev Container vs Local Setup vs venv

Feature Local Setup venv only Dev Container
Isolates Python packages ✗ ✓ ✓
Isolates system tools ✗ ✗ ✓
Same on every machine ✗ ✗ ✓
Defined as version-controlled code ✗ ✗ ✓
Works in GitHub Codespaces ✗ ✗ ✓
Key insight: Each project gets its own container. Two projects on the same machine are completely isolated — no dependency conflicts, no version clashes.
02

Prerequisites & Installation

You need three things installed on your host machine before you can use Dev Containers.

1 — Docker Desktop

Download from docker.com/products/docker-desktop. During installation on Windows, use these settings:

✓ Use WSL 2 instead of Hyper-V — faster, uses less memory, recommended.
□ Allow Windows Containers — leave unchecked. You will use Linux containers.
□ Add shortcut to desktop — personal preference only.

2 — VS Code

Download from code.visualstudio.com. Any recent version works.

3 — Dev Containers Extension

Install the Dev Containers extension by Microsoft inside VS Code (extension ID: ms-vscode-remote.remote-containers).

Verify Docker is running

terminal
$ docker --version
expected output
Docker version 24.0.7, build afdd53b
Docker daemon not running? You must open Docker Desktop and wait for it to fully start before VS Code can use it. Look for the Docker icon in your taskbar to stop animating.
03

Project Structure

All Dev Container config lives in a .devcontainer/ folder at the root of your project. This folder is committed to version control — it's what makes the environment portable.

Minimal structure

my-project/ ├── .devcontainer/ │ └── devcontainer.json ← main config file ├── requirements.txt ← Python dependencies ├── .gitignore └── main.py

Extended structure (with Dockerfile)

my-project/ ├── .devcontainer/ │ ├── devcontainer.json │ └── Dockerfile ← custom image definition ├── requirements.txt └── src/

How to generate it in VS Code

Press Ctrl + Shift + P → type Dev Containers: Add Dev Container Configuration Files → select Workspace → choose a template (e.g. Python 3) → select version → press OK.

Always choose Workspace — not user data folder. Workspace places the config inside your project so it gets committed to Git and shared with teammates.
04

devcontainer.json Explained

This is the heart of your Dev Container. Every setting that defines your environment goes here.

Example A — pre-built image (simplest)

.devcontainer/devcontainer.json
// devcontainer.json { "name": "Python 3", "image": "mcr.microsoft.com/devcontainers/python:3.11", "forwardPorts": [8000], "postCreateCommand": "pip3 install -r requirements.txt" }

Example B — custom Dockerfile

.devcontainer/Dockerfile
FROM mcr.microsoft.com/devcontainers/python:3.11 # Install extra system packages RUN apt-get update && apt-get install -y \ postgresql-client \ && rm -rf /var/lib/apt/lists/* # Install Poetry RUN pip install poetry
.devcontainer/devcontainer.json — with Dockerfile
{ "name": "Python + Postgres", "build": { "dockerfile": "Dockerfile" }, "forwardPorts": [8000, 5432], "postCreateCommand": "poetry install", "remoteUser": "vscode" }

Key properties reference

Property Purpose Example value
name Display name shown in VS Code title bar "My Project"
image Pre-built Docker image to use as base "python:3.11"
build Build from a local Dockerfile instead {"dockerfile": "Dockerfile"}
forwardPorts Expose container ports to your host machine [3000, 8080]
postCreateCommand Command to run once after container is created "pip install -r requirements.txt"
remoteUser User to run as inside the container "vscode"
features Add pre-packaged tools without a Dockerfile {"ghcr.io/.../node:1": {}}
customizations VS Code extensions and settings for the container {"vscode": {"extensions": [...]}}
Pin your image versions! Use python:3.11 not python:latest. Latest can change and break your environment unexpectedly.
05

Adding Features

Features are pre-packaged tool installers that you add directly to devcontainer.json. They are the cleanest way to add tools without writing a custom Dockerfile.

Example — Node + Python + Docker-in-Docker + GitHub CLI

.devcontainer/devcontainer.json
{ "name": "Multi-tool Dev", "image": "mcr.microsoft.com/devcontainers/base:ubuntu", "features": { "ghcr.io/devcontainers/features/node:1": { "version": "20" }, "ghcr.io/devcontainers/features/python:1": { "version": "3.11" }, "ghcr.io/devcontainers/features/docker-in-docker:2": {}, "ghcr.io/devcontainers/features/github-cli:1": {} } }

After opening in container, verify the tools are available:

terminal — verify installed features
$ node --version $ python3 --version $ gh --version
expected output
v20.11.0 Python 3.11.7 gh version 2.43.1 (2024-01-31)
Browse all official features at containers.dev/features — there are features for Go, Rust, Java, kubectl, Terraform, and many more.
06

Lifecycle Hooks

Dev Containers provide hooks that let you run commands at specific points in the container's lifecycle. Understanding which hook to use avoids wasted rebuild time.

Hook execution order

initializeCommand
→
onCreateCommand
→
postCreateCommand
→
postStartCommand
→
postAttachCommand

When each hook runs

Hook Runs on host? Runs every start? Common use
initializeCommand Yes (host) Yes Check prerequisites before container starts
onCreateCommand No (container) No — first create only One-time setup tasks
postCreateCommand No (container) No — first create only Install dependencies, run migrations
postStartCommand No (container) Yes Start background services
postAttachCommand No (container) Yes Display welcome message

Example configuration

.devcontainer/devcontainer.json — lifecycle hooks
{ "name": "My App", "image": "mcr.microsoft.com/devcontainers/python:3.11", "postCreateCommand": "pip install -r requirements.txt && python manage.py migrate", "postStartCommand": "service nginx start", "postAttachCommand": "echo '✅ Environment ready! Run: python main.py'" }
expected output when attaching
Running postCreateCommand: pip install -r requirements.txt... Successfully installed numpy-2.4.4 plotly-5.18.0 dash-2.16.1 Running migrations... ✅ Environment ready! Run: python main.py
07

Docker Compose Setup

When your project needs multiple services — like a database or cache alongside your app — use Docker Compose. Your editor connects to the app service while other services run alongside it.

docker-compose.yml

.devcontainer/docker-compose.yml
version: '3.8' services: app: build: context: . dockerfile: Dockerfile volumes: - ..:/workspace:cached command: sleep infinity # keeps container alive db: image: postgres:15 environment: POSTGRES_PASSWORD: devpassword POSTGRES_DB: myapp_dev redis: image: redis:7-alpine

devcontainer.json for Compose

.devcontainer/devcontainer.json
{ "name": "Full Stack Dev", "dockerComposeFile": "docker-compose.yml", "service": "app", "workspaceFolder": "/workspace", "forwardPorts": [8000, 5432, 6379], "postCreateCommand": "pip install -r requirements.txt" }
expected output when container starts
[1 of 3] Starting app ... [2 of 3] Starting db ... done [3 of 3] Starting redis ... done Running postCreateCommand: pip install -r requirements.txt ✓ All packages installed. Container ready.
08

Environment Variables & Secrets

Never hardcode secrets in devcontainer.json. The file is committed to version control and visible to everyone. Use these patterns instead.

Pattern A — Read from host environment

devcontainer.json — reading host env vars
{ "remoteEnv": { "DATABASE_URL": "${localEnv:DATABASE_URL}", "API_KEY": "${localEnv:MY_API_KEY}" } }

The ${localEnv:VAR_NAME} syntax reads a variable from your host machine — nothing sensitive enters the config file.

Pattern B — Local .env file (gitignored)

devcontainer.json — using .env.local file
{ "runArgs": ["--env-file", ".devcontainer/.env.local"] }
.gitignore — exclude the secrets file
# Dev Container secrets — never commit this .devcontainer/.env.local
Never commit .env files. Add them to .gitignore immediately. Use a .env.example file with placeholder values to document what variables are needed.
09

Best Practices

✅ Do

Pin image versions. Use python:3.11 not python:latest — latest changes and breaks reproducibility.
Use Features over custom Dockerfiles when possible. Features are maintained by the community, composable, and require zero Dockerfile knowledge.
Cache dependencies with named volumes to avoid reinstalling packages on every rebuild:
devcontainer.json — volume mount for dependency cache
{ "mounts": [ "source=myproject-pip-cache,target=/root/.cache/pip,type=volume" ] }
Set remoteUser to a non-root user. Most Microsoft base images use vscode — always prefer this over running as root to avoid file permission issues.
Always commit .devcontainer/ to version control. That is the entire point — sharing the environment definition with your team.

❌ Avoid

Do not run as root inside the container. It causes file ownership problems and is a security risk.
Do not hardcode secrets. API keys, passwords, and tokens must never appear in devcontainer.json or any committed file.
Do not make images too heavy. A container that takes 15 minutes to build discourages use. Add only what the project actually needs.
Do not skip .dockerignore. Without it, Docker sends your entire project directory as build context — including node_modules, .venv, and other large folders — slowing every rebuild.

Recommended .gitignore entries

.gitignore
# Python __pycache__/ *.pyc *.pyo .venv/ venv/ # Secrets .env .env.local .devcontainer/.env.local # OS .DS_Store
