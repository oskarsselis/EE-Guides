00

WSL setup (Windows only)

WSL — Windows Subsystem for Linux — lets you run a real Linux environment directly inside Windows, without a virtual machine. For firmware projects like Betaflight, which rely on Linux-native build tools, WSL is the recommended way to work on Windows. All subsequent steps in this guide are run inside WSL.

Requirement: Windows 10 version 2004 or later, or Windows 11. Run all commands below in PowerShell or Command Prompt opened as Administrator.

Install WSL with Ubuntu

A single command installs WSL and the default Ubuntu distribution in one step.

PowerShell — run as administrator
PS> wsl --install
expected output
Installing: Virtual Machine Platform Virtual Machine Platform has been installed. Installing: Windows Subsystem for Linux Windows Subsystem for Linux has been installed. Installing: Ubuntu Ubuntu has been installed. The requested operation is successful. Please restart your machine to complete the installation.

Restart your computer after this completes. When Windows boots back up, an Ubuntu terminal window will open automatically and finish setup — you'll be asked to create a Linux username and password. These are separate from your Windows credentials.

Note: The Linux password you set here is used whenever you run sudo commands inside WSL. Choose something you'll remember — it does not need to match your Windows password.

Check WSL status

After restarting, open PowerShell and verify WSL is running correctly and shows your Ubuntu instance.

PowerShell
PS> wsl --status
expected output
Default Distribution: Ubuntu Default Version: 2 Windows Subsystem for Linux was last updated on ... The Windows Subsystem for Linux kernel can be manually updated ...

Confirm that Default Version: 2 is shown. WSL 2 uses a real Linux kernel and is significantly faster than WSL 1 for build-heavy projects like Betaflight.

Check WSL version and list installed distributions

This shows every installed Linux distribution and confirms which version of WSL each one is running on.

PowerShell
PS> wsl --list --verbose
expected output
NAME STATE VERSION * Ubuntu Running 2

The * marks the default distribution. VERSION 2 confirms WSL 2 is active. If you see VERSION 1, upgrade it with wsl --set-version Ubuntu 2.

Update and upgrade Ubuntu packages

Open the Ubuntu app from the Start menu (or type wsl in PowerShell to enter it). Run both commands every time you set up a fresh Ubuntu instance — this ensures all system packages are current before you install anything else.

Ubuntu (WSL terminal)
$ sudo apt update $ sudo apt upgrade -y
expected output (apt update)
Hit:1 http://archive.ubuntu.com/ubuntu jammy InRelease Get:2 http://archive.ubuntu.com/ubuntu jammy-updates InRelease [128 kB] Fetched 3,241 kB in 4s (810 kB/s) Reading package lists... Done Building dependency tree... Done
expected output (apt upgrade)
Reading package lists... Done Building dependency tree... Done Calculating upgrade... Done 0 upgraded, 0 newly installed, 0 to remove and 0 not changed. (or: N upgraded, N newly installed ...)

apt update refreshes the list of available packages and their versions — it does not install anything. apt upgrade -y then downloads and installs any newer versions of packages already on your system. The -y flag automatically answers "yes" to all prompts so the upgrade runs unattended.

From here on: All terminal commands in this guide are run inside the Ubuntu WSL terminal, not in PowerShell. Open the Ubuntu app from your Start menu or type wsl in any PowerShell window to enter it.
01

Install & configure Git

Git is the version control tool that runs on your computer. GitHub is the website that hosts repositories. They are separate things — Git works without GitHub, but you need Git installed to use GitHub from the command line.

Install

Windows — download the installer from git-scm.com and run it with default options.
macOS — run xcode-select --install in Terminal. Git is bundled with Xcode tools.
Linux (Debian/Ubuntu) — use your package manager:

terminal
$ sudo apt install git

Verify installation

Check that Git is available and see its version number.

terminal
$ git --version
expected output
git version 2.44.0

Any version above 2.30 is fine. If you see command not found, the install did not complete — try again.

Set your identity

Git attaches your name and email to every commit. This must match your GitHub account so your contributions are linked to your profile.

terminal
$ git config --global user.name "Your Name" $ git config --global user.email "[email protected]"

--global means this setting applies to every repository on your machine. You can override it per-project with --local inside a specific repo folder.

Verify your config

terminal
$ git config --list
expected output (excerpt)
core.editor=vim user.name=Your Name [email protected] core.autocrlf=input

Confirm user.name and user.email appear with the values you set.

02

Fork the repository

A fork is your own personal copy of someone else's repository, stored on GitHub under your account. You cannot push changes directly to betaflight/betaflight — you don't have write permission. A fork gives you a repository that you fully own.

This step is done entirely in the browser — no terminal needed:

  1. Go to github.com/betaflight/betaflight
  2. Click the Fork button in the top-right corner
  3. Select your own GitHub account as the destination
  4. Click Create fork

You now have github.com/YOUR_USERNAME/betaflight — this is fully yours. All your work will go here first before you propose it to the official project.

Tip: The fork stays permanently linked to the original repo. GitHub shows "forked from betaflight/betaflight" on your fork's page, and this link is what enables pull requests later.
03

Clone your fork

Cloning downloads the entire repository — all files and the full commit history — to your computer. You clone your fork, not the official repo, so you have write access.

Navigate to your home directory first

Always clone into your home directory rather than a random location. This keeps all your projects in a predictable, consistent place and avoids permission issues — especially important inside WSL where the Linux home directory (~) is separate from your Windows files.

terminal
$ cd ~ $ pwd
expected output
/home/your_username

cd ~ takes you to your Linux home directory regardless of where you currently are. pwd (print working directory) confirms the exact path — use this any time you want to know where you are in the filesystem.

Clone the repository

terminal
$ git clone --recursive https://github.com/YOUR_USERNAME/betaflight.git $ cd betaflight
expected output
Cloning into 'betaflight'... remote: Enumerating objects: 84321, done. remote: Counting objects: 100% (84321/84321), done. Receiving objects: 100% (84321/84321), 68.42 MiB | 4.2 MiB/s, done. Resolving deltas: 100% (62148/62148), done. Submodule 'lib/main/STM32F4' (https://github.com/betaflight/STM32F4.git) registered Cloning into '/home/your_username/betaflight/lib/main/STM32F4'... Submodule path 'lib/main/STM32F4': checked out 'a7df4b...'

git clone <url> creates a new folder named after the repo and downloads everything into it. The --recursive flag also initialises and downloads any submodules — these are separate Git repositories that the main project depends on, referenced at a specific commit. Betaflight uses submodules for hardware libraries (e.g. STM32 peripheral drivers). Without --recursive those folders would be empty and the build would fail.

cd betaflight moves your terminal into the cloned folder. All future Git commands must be run from inside a repository folder.

Forgot --recursive? If you already cloned without the flag, you can recover the missing submodules at any time. See Step 9 — submodule update for the full command and expected output.

Verify the remote

Check which remote servers Git knows about after cloning.

terminal
$ git remote -v
expected output
origin https://github.com/YOUR_USERNAME/betaflight.git (fetch) origin https://github.com/YOUR_USERNAME/betaflight.git (push)

origin is the default name Git assigns to the remote you cloned from. At this point, only your fork is registered — the official Betaflight repo is not yet connected.

04

Add the upstream remote

You need a connection to the official Betaflight repository so you can pull in future updates. By convention, this remote is called upstream.

terminal
$ git remote add upstream https://github.com/betaflight/betaflight.git

Verify both remotes

terminal
$ git remote -v
expected output
origin https://github.com/YOUR_USERNAME/betaflight.git (fetch) origin https://github.com/YOUR_USERNAME/betaflight.git (push) upstream https://github.com/betaflight/betaflight.git (fetch) upstream https://github.com/betaflight/betaflight.git (push)
Remote Points to You can push to it? Used for
origin your fork on GitHub Yes Pushing your branches and PRs
upstream official betaflight repo No Fetching official updates

Find the default branch name

Different projects use different names for their main branch (master, main, develop, etc.). Betaflight uses master. Always verify before assuming.

terminal
$ git branch -r
expected output
origin/HEAD -> origin/master origin/master upstream/master

The -r flag lists remote-tracking branches. The line origin/HEAD -> origin/master tells you the default branch is master.

05

Create a feature branch

Never make changes directly on master. A branch is an isolated copy of the codebase where your changes live. If something goes wrong, the rest of your code is untouched.

terminal
$ git checkout -b fpvfc_h753_2025.12.2
expected output
Switched to a new branch 'fpvfc_h753_2025.12.2'

git checkout -b <name> creates a new branch and switches to it in one step. Without the -b flag, checkout just switches to an existing branch — the flag is what creates it.

Use a short descriptive name. Common prefixes:

  • fix/ — bug fixes
  • feature/ — new functionality
  • docs/ — documentation changes

Verify you're on the new branch

terminal
$ git branch
expected output
* fpvfc_h753_2025.12.2 master

The * marks your currently active branch. This confirms you are working in isolation from master.

06

Edit, stage & commit

After you edit files in your editor, Git tracks what changed but won't save it permanently until you create a commit. This is a two-step process: first stage the files you want to include, then commit them.

Check what changed

terminal
$ git status
expected output
On branch fpvfc_h753_2025.12.2 Changes not staged for commit: modified: src/main/flight/mixer.c Untracked files: src/main/flight/mixer_test.c

Stage your changes

Staging is a preparatory step — you're telling Git "include this in the next commit." You have not committed anything yet.

terminal — stage a specific file
$ git add src/main/flight/mixer.c
Caution: git add . stages every changed file at once. Be careful — you may accidentally include files you didn't intend to commit, such as local config files or build outputs.

Commit with a message

terminal
$ git commit -m "fix: correct motor output scaling in mixer"
expected output
[fpvfc_h753_2025.12.2 a3f9c12] fix: correct motor output scaling in mixer 1 file changed, 8 insertions(+), 3 deletions(-)

The short hash (e.g. a3f9c12) is Git's unique ID for this commit. The fix: prefix follows the Conventional Commits convention — many projects including Betaflight encourage this format.

Verify the commit was saved

terminal
$ git log --oneline -5
expected output
a3f9c12 fix: correct motor output scaling in mixer 8e201bd feat: add blackbox logging for gyro d14a990 refactor: clean up PID loop timing c32f881 fix: resolve OSD font rendering glitch b90e112 docs: update build instructions for macOS

Your commit appears at the top of the log with its hash and message.

07

Push to your fork

Your commit currently exists only on your local machine. Pushing sends it to your fork on GitHub, making it available online and enabling you to open a pull request.

terminal
$ git push origin fpvfc_h753_2025.12.2
expected output
Enumerating objects: 7, done. Counting objects: 100% (7/7), done. Delta compression using up to 8 threads Compressing objects: 100% (4/4), done. Writing objects: 100% (4/4), 1.12 KiB | 1.12 MiB/s, done. To https://github.com/YOUR_USERNAME/betaflight.git * [new branch] fpvfc_h753_2025.12.2 -> fpvfc_h753_2025.12.2

Breaking down the command: push uploads commits, origin is the remote to push to (your fork), and fpvfc_h753_2025.12.2 is the branch name to create or update on the remote.

Verify: Go to github.com/YOUR_USERNAME/betaflight in your browser. You'll see a yellow banner saying "fpvfc_h753_2025.12.2 had recent pushes" with a button to compare and open a pull request.
08

Open a pull request

A Pull Request (PR) is a formal proposal asking the Betaflight maintainers to review and merge your changes into their repository. This step is done in the browser.

  1. Click Compare & pull request from the yellow banner on your fork, or go to the Pull Requests tab and click New pull request
  2. Set the base to betaflight/betaflight : master
  3. Set the compare to your branch YOUR_USERNAME/betaflight : fpvfc_h753_2025.12.2
  4. Write a clear title: fix: correct motor output scaling in mixer
  5. In the description, explain what changed, why it was needed, and how you tested it
  6. Click Create pull request
What happens next: Maintainers will review your code. They may request changes — you can push additional commits to the same branch and the PR updates automatically. Eventually it will either be merged or closed with feedback.
09

Sync with upstream

While you work, the official Betaflight repo keeps moving. You should regularly sync your local master and your fork to stay current. There are three separate locations involved — each command touches a specific pair of them.

Location What it is Touched by
upstream/master Official betaflight repo on GitHub git fetch upstream
local master Your machine's master branch git rebase upstream/master
origin/master Your fork on GitHub git push origin master
feature branch Your work branch Not touched by sync steps

Step 9a — fetch from upstream

Downloads the latest commits from the official repo to a hidden local reference called upstream/master. This is the safest command in Git — nothing on disk changes, no files are modified, no branches move. It only makes your computer aware of what exists upstream.

terminal
$ git fetch upstream
expected output (if updates exist)
remote: Enumerating objects: 48, done. remote: Counting objects: 100% (48/48), done. From https://github.com/betaflight/betaflight 8e201bd..f32c918 master -> upstream/master

The line 8e201bd..f32c918 master -> upstream/master tells you new commits arrived. If there's no output, you're already up to date.

terminal — verify: see upstream's latest commits
$ git log upstream/master --oneline -5

Fetching tags from upstream

A regular git fetch upstream downloads commits and branch pointers but does not automatically download tags. Tags are used by projects like Betaflight to mark release versions (e.g. 4.4.3, 4.5.0). Use the --tags flag to fetch them explicitly.

terminal
$ git fetch upstream --tags
expected output (if new tags exist)
From https://github.com/betaflight/betaflight * [new tag] 4.5.0 -> 4.5.0 * [new tag] 4.5.1 -> 4.5.1

Once fetched, list all tags in the repo to see available release versions:

terminal
$ git tag -l
expected output
4.3.0 4.3.1 4.4.0 4.5.0 4.5.1

-l lists all tags alphabetically. You can filter with a pattern, for example git tag -l "4.5.*" lists only 4.5.x releases. Tags are useful for checking out the exact state of the code at a specific release: git checkout 4.5.0.

Step 9b — switch to master and rebase

You must be on your local master branch before rebasing it — otherwise you'd rebase your feature branch by mistake. Rebase replays your local master on top of the new upstream commits. In practice, since you haven't committed directly to master yourself, this simply moves the master pointer forward.

terminal
$ git checkout master $ git rebase upstream/master
expected output
Switched to branch 'master' Your branch is behind 'origin/master' by 12 commits, and can be fast-forwarded. Successfully rebased and updated refs/heads/master.
Command What it does Creates a merge commit?
git rebase upstream/master Replays commits on top of the new base. Linear history. No — clean history
git merge upstream/master Joins histories with a merge commit. Yes — extra commit

Most open-source projects, including Betaflight, prefer rebase for keeping forks in sync because it produces cleaner, linear history.

Step 9c — push updated master to your fork

At this point, your local master is synced but your fork on GitHub is still outdated. This is the only command in the sequence that reaches the internet and modifies your fork.

terminal
$ git push origin master
expected output
To https://github.com/YOUR_USERNAME/betaflight.git 8e201bd..f32c918 master -> master

Optional — rebase your feature branch onto the synced master

Do this before opening a PR. It brings the new upstream changes into your feature branch so your work is based on the latest code, making it much easier for maintainers to review and merge.

terminal
$ git checkout fpvfc_h753_2025.12.2 $ git rebase master
expected output
Switched to branch 'fpvfc_h753_2025.12.2' Successfully rebased and updated refs/heads/fpvfc_h753_2025.12.2.
Conflict? If the same lines were changed both upstream and in your branch, Git will pause and ask you to resolve the conflict manually. Open the affected file, find the conflict markers (<<<<<<< / ======= / >>>>>>>), edit the file to the correct state, then run git add <file> and git rebase --continue.

Step 9d — update submodules after syncing

After rebasing onto a newer upstream, the submodule pointers in the repo may have changed — the official Betaflight master may now reference a newer commit of an STM32 library than you had locally. Running git submodule update brings every submodule in sync with what the current branch expects. This is also the recovery command if you originally cloned without --recursive.

terminal
$ git submodule update --init --recursive
expected output
Submodule 'lib/main/STM32F4' (https://github.com/betaflight/STM32F4.git) registered Cloning into '/home/your_username/betaflight/lib/main/STM32F4'... Submodule path 'lib/main/STM32F4': checked out 'a7df4b...'

--init registers any submodules that are declared in the repo but haven't been initialised yet. --recursive descends into nested submodules (submodules that themselves contain submodules). If everything was already up to date, Git exits silently with no output.

Quick reference

Setup & inspection

git init Init a new repo in current folder
git status See what changed
git log --oneline Browse commit history
git remote -v List configured remotes
git tag -l List all tags
git submodule update --init --recursive Init & fetch all submodules

Branches

git branch List local branches
git branch -r List remote branches
git checkout -b <name> Create and switch to branch
git checkout <name> Switch to existing branch
git branch -M <new-name> Force-rename current branch
git branch --set-upstream-to=origin/<branch> Set remote tracking branch

Stage, commit & cherry-pick

git add <file> Stage a specific file
git commit -m "msg" Commit staged changes
git cherry-pick <hash> Apply one commit to current branch

Push, fetch & sync

git push origin <branch> Push branch to your fork
git push origin :<branch> Delete a remote branch
git fetch upstream Download upstream commits
git fetch upstream --tags Download upstream tags too
git rebase upstream/master Sync local master
